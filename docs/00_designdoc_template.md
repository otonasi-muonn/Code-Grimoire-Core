# 設計メモ：Code-Grimoire リメイク（プラグインアーキテクチャによる描画・解析の分離）

---

## 1. 何を解決したい？
- **困っていること（いま何が問題？）**：
  - 既存の `Code-Grimoire` はTS/JSの解析と描画が密結合しており、他言語（Python, Go, Rustなど）への対応が非常に困難。
  - プロジェクトのフォルダ構成や内部設計が肥大化・複雑化し、メンテナンスや新機能追加がしづらくなっている。
  - 自分ひとりで全言語のパーサーを書くのは現実的ではない。
- **既存サービスを使った解決策とその問題点**：
  - 既存のコード可視化ツールは静的な画像出力が多く、インタラクティブで「美しい（中二病心をくすぐる）」体験を提供するものがない。
  - 既存の `Code-Grimoire` をそのまま拡張しようとすると、スパゲッティコード化（Big Ball of Mud）が避けられない。
- **だれが困る？**：
  - Code-Grimoireの可視化を他言語のプロジェクトでも使いたいユーザー。
  - 新しい機能や言語対応を追加したい開発者（自分自身および将来のコントリビューター）。
- **うまくいったと言える条件（できれば数字）**：
  - 解析（バックエンド）と描画（フロントエンド）のリポジトリが完全に分離されている。
  - Core側が一切TS/JSのコード（AST等）を知らなくても、ダミーデータを渡すだけで魔方陣が描画できる。
  - 第1弾のプラグイン（例: `code-grimoire-ts`）がCoreと連携して正常に動作する。

---

## 2. 今回やらないこと
- **やらないこと**：
  - **Coreリポジトリ内での具体的な言語解析（AST解析など）の実装**。
  - VS CodeのLanguage Server Protocol (LSP) を直接フックする高度な解析（将来の展望とし、最初は独自のJSONスキーマ渡しとする）。
  - 最初から複雑なレイアウト（銀河、泡宇宙）の完全実装（まずは魔方陣のForceレイアウトの再現を優先）。

---

## 3. アイデア
- **こうする**：
  - システムを「描画コア（Code-Grimoire-Core）」と「言語別解析プラグイン」に分割する（クリーンアーキテクチャ/プラグインアーキテクチャの適用）。
  - CoreはVS CodeのExtension API (`exports`) を使って、外部からノードとエッジのデータを受け取る口（インターフェース）だけを公開する。
  - 描画には引き続き PixiJS (WebGL) と d3-force (Web Worker) を使い、圧倒的なパフォーマンスを維持する。
- **なぜそうする？（理由は2〜3個まで）**：
  1. **開放閉鎖の原則（OCP）**：Coreを修正することなく、他の開発者が独自の言語プラグインを作成してエコシステムを拡張できるようにするため。
  2. **単一責任の原則（SRP）**：UI/描画の責務と、コード解析のドメインロジックの責務を完全に切り離し、メンテナンス性を劇的に向上させるため。

---

## 4. 他の案
- **案A：既存のCode-Grimoireをリファクタリングして多言語対応する**
  - 良いところ：既存のコード資産をそのまま流用できる。
  - 悪いところ：アーキテクチャの根本的な解決にならず、言語が増えるたびにCoreのコードベースが肥大化する。
- **今回選ばない理由（短く）**：
  - コミュニティ主導で言語対応をスケールさせる（他者にパーサーを作ってもらう）という最大の目標が達成できないため。

---

## 5. 導入によって生じるエコシステム・ユーザーの変化
- **フローの変化**：
  - Before: ユーザーは1つの拡張機能を入れるだけだが、TS/JSしか対応していない。開発者は1つのリポジトリをいじる。
  - After: ユーザーは「Core」と「自分の使いたい言語のプラグイン」を入れる。開発者は各言語ごとに独立して開発できる。
- **全体での変化(拡張性/コミュニティ面)**：
  - Before: 作者（自分）が全ての言語仕様を把握し、実装・保守しなければならない。
  - After: 共通APIスキーマの仕様さえ満たせば、世界中の誰でも任意の言語の「魔導書（プラグイン）」を発行できる。
- **利用者個人の変化(心理的側面)**：
  - Before: 「自分の言語でも使えたらいいのに」と諦める。
  - After: プラグインさえ作れば/入れれば、どんな言語・フレームワークでも美しい魔方陣として可視化できる感動を得られる。

### 6.1 全体の見積もり（ざっくりでOK）
- **全体の作業時間**：〇〇時間 / 〇週間
- **使える日**：平日〇時間 + 土日〇時間

### 6.2 最初にやること（1〜3日で終わるサイズ）
- **まず作る最小版（MVP）**：
  - `Code-Grimoire-Core` にPixiJSとd3-forceの基盤をセットアップする。
  - 拡張機能の `activate` 内で適当なモックデータ（NodeとEdgeの配列）を生成し、Webviewに送って描画させる。
- **それで確かめたいこと**：
  - 新しい環境（Vite/esbuildなど）でPixiJSとWeb Workerが正しく連携し、パフォーマンスが出るか。
- **できなかったらどうする（簡単な逃げ道）**：
  - Web Workerの連携で詰まったら、一旦メインスレッドでd3-forceを動かして描画を優先する。
